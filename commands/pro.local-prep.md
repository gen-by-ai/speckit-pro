# SpecKit Pro — Local Prep (`pro.local-prep`)

Runs **local Ollama workers** to produce the Markdown prep artifacts that a Claude-driven loop would otherwise generate inline (and pay tokens for). Claims in these files are **drafts** — Claude verifies them downstream.

This command is fast to skip and slow to abuse. If `local_models.enabled: false` or Ollama is unreachable, the command prints a one-line note and exits 0. It NEVER aborts the parent pipeline.

## When this runs

- Manual: `/speckit.pro.local-prep [--spec-dir <path>] [--only <list>] [--force]`
- Pipeline: `pro.go` Phase 4b — automatically after `/speckit.tasks` if `local_models.enabled: true` and `local_models.auto_run.after_tasks: true`.
- Hook: `after_tasks` (optional, opt-in).

## What it writes

Under `<SPEC_DIR>/`:

| File                  | Purpose                                                                | Read by         |
|-----------------------|------------------------------------------------------------------------|-----------------|
| `repo-map.md`         | Relevant files, patterns, test commands, risks, unknowns               | Implementer     |
| `risk-register.md`    | Concrete risks with triggers, severity, mitigation, verifier           | Implementer + evaluator |
| `test-strategy.md`    | Commands and case ideas grounded in the project's CI                   | Implementer     |
| `open-questions.md`   | ≤ 10 focused human-input questions, multiple-choice when possible      | Operator        |
| `context-pack.md`     | Compiled, ≤ 1500-word context bundle for the next iteration            | Implementer + evaluator |

Each file is prepended with a provenance banner: `Generated by local model <name> via ollama-md.py. Claims require verification before implementation.` That banner is intentional — it stops a downstream agent (or a human) from treating local-model output as ground truth.

## User Input

```text
$ARGUMENTS
```

Optional flags:
- `--spec-dir <path>` — target a specific feature directory; auto-detected otherwise.
- `--only <list>` — comma-separated subset of `repo-map,context-pack,risk-register,test-strategy,open-questions`.
- `--force` — regenerate even if the file already exists.
- `--dry-run` — print what would happen without calling Ollama.

## Steps

### 1. Resolve target feature

Run:
```bash
PROJECT_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || pwd)
```
If `--spec-dir` is not provided, the script auto-detects via `.specify/scripts/bash/check-prerequisites.sh --json`, falling back to the most-recently-modified `specs/*/spec.md`.

### 2. Check the local-models config

The driver loads `local_models.*` from `.specify/extensions/pro/pro-config.yml` (or local override). If `local_models.enabled` is `false`, print:
```
[local] local_models.enabled is false — skipping local prep.
        Set local_models.enabled: true in pro-config.yml to opt in.
```
and exit 0.

### 3. Check Ollama reachability

A 3-second `curl /api/tags` probe. If unreachable, print:
```
[local WARN] Ollama not reachable at <base_url> — skipping local prep.
             Start it:   ollama serve
             Or disable: local_models.enabled: false
```
and exit 0.

### 4. Run the driver

Execute:
```bash
bash <EXTENSION_ROOT>/scripts/bash/pro-local-prep.sh \
  --spec-dir <SPEC_DIR> \
  [--only "<list>"] [--force] [--dry-run]
```

The driver:
1. Builds a shallow file-tree snapshot (≤ 500 paths via `git ls-files`) — grounds `repo-map.md` so the model doesn't hallucinate paths.
2. Generates `repo-map.md` first (cheap; downstream artifacts feed on it).
3. Generates `risk-register.md`, `test-strategy.md`, `open-questions.md` in parallel-safe order (serial by default — concurrency 1 per dev.md guidance for Pi-class hardware).
4. Compiles `context-pack.md` last, fed by everything above plus `.knowledge/INDEX.md` if present.

### 5. Reading the output

The provenance banner makes it visible in PR review that these files are drafts. Edit freely. The next time `/speckit.implement` runs, the loop will read `context-pack.md` (lean) instead of spec + plan + tasks (full).

## What this command does NOT do

- It does not run the implementation loop. That's still `/pro.go` → Phase 6.
- It does not silently mutate spec.md, plan.md, or tasks.md. Those remain hand-authored.
- It does not commit. Run `/pro.checkpoint` when you're ready to snapshot.
- It does not use Claude. By design — this is the "stop using Claude for janitorial work" command.

## Configuration knobs (excerpt from `pro-config.yml`)

```yaml
local_models:
  enabled: false                      # master switch (opt-in)
  base_url: "http://localhost:11434"
  default_model: "qwen2.5-coder:7b"
  fast_model:    "llama3.2:3b"
  code_model:    "qwen2.5-coder:7b"
  review_model:  "qwen2.5-coder:7b"
  security_model: "qwen2.5-coder:7b"
  timeout_seconds: 180
  num_ctx: 8192
  temperature: 0.2
  tasks:
    repo_map: true
    context_pack: true
    risk_register: true
    test_strategy: true
    open_questions: true
  auto_run:
    after_tasks: true                 # run automatically from pro.go Phase 4b
    before_evaluate: true             # pro.local-review runs in Phase 6b
```

## Remote Ollama

For Pi-class hardware that can't run a 7B model locally, point at a workstation:
```yaml
local_models:
  base_url: "http://workstation.local:11434"
```
or export `OLLAMA_BASE_URL` in the environment. The driver uses whichever resolves first.

## Why this exists

From `.dev-work/dev.md`:

> The Claude extension is good for interactive work, but your target is closer to a workflow engine: queue + artifacts + model router + workers + gates. Claude should become a premium worker, not the whole factory.

`pro.local-prep` moves the highest-token / lowest-judgement prep work to local models. Token savings compound: every saved prep call is a Claude turn that goes to actual implementation instead.
